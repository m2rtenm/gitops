# Fraud Detection Service

Performs real-time fraud detection and alerting on all transactions. This service analyzes transaction patterns and flags suspicious activity for review.

## Integration with Other Services

- **Transaction Service**: Consumes all transactions for analysis
- **Ledger Service**: Reads account history and patterns
- **Settlement Service**: Blocks settlement until fraud cleared
- **Account Service**: Provides account holder information for context

## API Endpoints

### Health & Status
- `GET /health` - Health check
- `GET /ready` - Readiness check
- `GET /metrics` - Prometheus metrics

### Fraud Detection
- `GET /api/v1/alerts` - List fraud alerts (paginated)
- `GET /api/v1/alerts/{id}` - Get alert details
- `PUT /api/v1/alerts/{id}` - Review/approve/reject alert
- `GET /api/v1/alerts/transaction/{transaction_id}` - Get alerts for transaction

### Analytics
- `GET /api/v1/analytics/high-risk` - Get high-risk transactions
- `GET /api/v1/analytics/patterns/{account_id}` - Analyze account patterns
- `GET /api/v1/analytics/summary` - Get fraud metrics summary

## Database Schema

```sql
-- Fraud alerts generated by detection rules
CREATE TABLE fraud_alerts (
  id BIGSERIAL PRIMARY KEY,
  transaction_id BIGINT NOT NULL,
  alert_type VARCHAR(50) NOT NULL,
  risk_score NUMERIC(5, 2) NOT NULL,
  alert_message TEXT,
  rule_name VARCHAR(255),
  created_at TIMESTAMP DEFAULT NOW(),
  reviewed_at TIMESTAMP,
  reviewed_by VARCHAR(255),
  action_taken VARCHAR(50)
);

-- Audit trail for all transaction processing
CREATE TABLE transaction_audit (
  id BIGSERIAL PRIMARY KEY,
  transaction_id BIGINT NOT NULL,
  account_id BIGINT NOT NULL,
  action VARCHAR(50) NOT NULL,
  timestamp TIMESTAMP DEFAULT NOW(),
  user_id VARCHAR(255),
  ip_address VARCHAR(45),
  status VARCHAR(20),
  error_message TEXT
);

CREATE INDEX idx_alerts_transaction ON fraud_alerts(transaction_id);
CREATE INDEX idx_alerts_risk_score ON fraud_alerts(risk_score);
CREATE INDEX idx_alerts_created ON fraud_alerts(created_at);
CREATE INDEX idx_audit_transaction ON transaction_audit(transaction_id);
CREATE INDEX idx_audit_account ON transaction_audit(account_id);
```

## Fraud Detection Rules

### Risk Scoring Rules

1. **Unusual Amount** (weight: 0.3)
   - Transaction > 3x account average: +0.2
   - Transaction > 10x account average: +0.5

2. **Velocity** (weight: 0.25)
   - > 5 transactions in 1 minute: +0.3
   - > 10 transactions in 1 hour: +0.2

3. **Geographic Anomaly** (weight: 0.2)
   - Transaction from new country: +0.15
   - Impossible travel distance/time: +0.4

4. **Account Behavior** (weight: 0.25)
   - After-hours activity (if unusual): +0.1
   - New payee: +0.05
   - Suspended account: +0.8

### Alert Actions

- **LOW** (0.0-0.3): Log only
- **MEDIUM** (0.3-0.7): Flag for review, hold settlement pending review
- **HIGH** (0.7-1.0): Auto-block, immediate review required

## Data Flow

```
Transaction Service publishes fintech.transactions
    ↓
Fraud Detection Service consumes
    ↓
Scores transaction against rules
    ↓
If risk_score > threshold:
  Creates fraud_alert
  Publishes fintech.fraud-alerts
  Settlement Service delays until reviewed
    ↓
Settlement Service can:
  - Proceed if alert approved
  - Block if alert rejected
  - Hold if still pending
```

## Alert Lifecycle

```
CREATED (risk_score calculated)
  ↓
PENDING (awaiting human review)
  ├→ APPROVED (transaction allowed)
  ├→ REJECTED (transaction blocked)
  └→ INVESTIGATION (escalated)
```

## Performance Considerations

- Real-time analysis on transaction stream
- Caching of account patterns for faster scoring
- Batch processing of low-risk transactions
- Alert indexing for fast review queue access
- Configurable thresholds for false positive tuning
